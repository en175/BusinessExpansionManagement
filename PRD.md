## 一、产品概述

### 1.1 背景与目标

广州仲裁委员会（广仲）全委共 196 名干部职工，日常开展走访、来访、研讨会等业务拓展活动，当前缺乏统一的信息化登记与管理机制，导致活动数据分散、审批流程不规范、统计分析困难。

本模块在现有「仲裁办案管理系统」中新增「业务拓展管理」专项模块，实现活动信息的结构化登记、部门审批、查询统计与导出。

**目标**：
- 规范全委业务拓展活动的登记与审批流程
- 沉淀开拓对象数据，支持后续与案件数据关联
- 为战略发展部提供全委活动的查询、统计与导出能力

### 1.2 用户角色与权限矩阵

本模块从甲方现有 SSO 系统读取用户身份信息，不在模块内维护独立的账号体系。角色由 SSO token 中的用户职务字段决定。

| 角色 | 判定条件 | 登记活动 | 我的记录 | 待我审批 | 查询与统计 |
|------|---------|:--------:|:--------:|:--------:|:----------:|
| 普通员工 | `role = staff`，且不属于战略发展部 | ✓ | ✓ | — | — |
| 部门正职 | `role = head` | ✓ | ✓ | ✓ | ✓ |
| 战略发展部成员 | 所属部门为战略发展部（含正职、副职、干部） | ✓ | ✓ | 仅正职可用 | ✓ |

> **说明**：「待我审批」仅对 `role = head` 的用户展示；战略发展部的副职和普通干部可访问「查询与统计」，其他部门的副职和干部不可访问。

**SSO 集成要求**：
- 登录后从 SSO 获取：`userId`、`name`、`dept`（所属部门名称）、`role`（`head` / `deputy` / `staff`）
- 模块根据上述字段动态决定可访问菜单项
- 当前访问页面因角色变更而不再可访问时，自动重定向至该角色的第一个可访问页

### 1.3 名词解释

| 术语 | 说明 |
|------|------|
| 活动记录 | 一次业务拓展活动的完整登记信息，是本模块的核心数据对象 |
| 开拓对象 | 本次活动拜访/接触的机构/企业，一条活动记录可包含多个开拓对象 |
| 填报人 | 发起并提交活动登记的员工，由系统从 SSO 自动读取，不可手动修改 |
| 审批人 | 填报人所在牵头部门的正职；若该部门无正职，则为第一副职 |
| 自动通过 | 填报人本身是部门正职时，提交即直接通过，系统自动记录审批信息 |
| 修改重提 | 被退回的记录经填报人修改后重新提交审批，原审批历史保留 |

## 二、功能模块详述

### 2.1 活动登记表单

**入口**：左侧导航「业务拓展活动登记」

**权限**：所有登录用户均可访问

**页面用途**：新建一条活动记录；或从「我的记录」点击「修改重提」时，预填原记录数据进行修改后重新提交。

#### 2.1.1 表单分区与字段规格

**第一区：基本信息**

| 字段 | 类型 | 必填 | 规则 |
|------|------|:----:|------|
| 活动标题 | 文本输入 | ✓ | 最多 50 字 |
| 填报人 | 只读文本 | 系统自动 | 从 SSO 读取当前用户姓名，不可编辑 |
| 所属部门 | 只读文本 | 系统自动 | 从 SSO 读取当前用户部门，不可编辑 |
| 我委对接人 | 下拉搜索 | ✓ | 从员工列表中选择，支持姓名模糊搜索，显示格式「姓名（部门）」 |
| 牵头部门 | 下拉单选 | ✓ | 从 16 个部门中选择；默认值为填报人所属部门 |
| 配合部门 | 下拉多选 | — | 从 16 个部门中多选，可为空 |

**第二区：活动信息**

| 字段 | 类型 | 必填 | 规则 |
|------|------|:----:|------|
| 活动时间 | 日期区间选择 | ✓ | 格式 YYYY-MM-DD，开始日期 ≤ 结束日期；单日活动两端相同 |
| 活动形式 | 下拉单选 | ✓ | 枚举：走访 / 来访 / 调研 / 合作洽谈 / 研讨会 / 宣讲会 / 其他活动形式 |
| 活动规模（人） | 数字输入 | ✓ | 整数，范围 1–9999 |

**第三区：参与人员**

| 字段 | 类型 | 必填 | 规则 |
|------|------|:----:|------|
| 我委参与人员 | 下拉多选 | ✓ | 从员工列表多选，支持姓名模糊搜索；至少选择 1 人 |
| 出席领导 | 下拉多选 | — | 从员工列表多选，可为空 |
| 对方参与人员 | 动态列表 | — | 每条包含：姓名、职务（如「XX公司法务总监」）、联系方式（手机/邮箱）；可动态添加/删除行；初始 1 行，可增至不限 |

**第四区：开拓对象信息**

可动态增减，至少保留 1 个对象，每个对象字段如下：

| 字段 | 类型 | 必填 | 规则 |
|------|------|:----:|------|
| 机构名称 | 文本输入 | ✓ | 填写机构/企业全称 |
| 所属行业 | 下拉单选 | ✓ | 枚举：金融 / 建设工程 / 房地产 / 文娱 / 体育 / 数字经济 / 知识产权 / 高端制造 / 航空航天 / 国际航运 / 低空经济 / 其他行业 |
| 所在省份 | 下拉单选 | ✓ | 省级行政区选择 |
| 所在城市 | 下拉单选 | ✓ | 联动省份，省份未选时禁用；切换省份时城市清空 |

**第五区：活动详情**

| 字段 | 类型 | 必填 | 规则 |
|------|------|:----:|------|
| 活动目的及成效 | 多行文本 | ✓ | 最多 1000 字，含发现的合作机会、对方诉求、达成的意向等 |
| 反馈意见建议 | 多行文本 | — | 最多 1000 字 |
| 后续跟进事项 | 多行文本 | — | 最多 1000 字 |

**第六区：附件与备注**

| 字段 | 类型 | 必填 | 规则 |
|------|------|:----:|------|
| 文件附件 | 文件上传 | — | 支持格式：PDF / DOC / DOCX / JPG / JPEG / PNG；单文件 ≤ 50 MB；可上传多个；支持上传后预览（图片直接预览，PDF/DOC 展示文件名） |
| 备注 | 多行文本 | — | 最多 500 字 |

#### 2.1.2 提交逻辑

1. 点击「提交审批」，触发全表单校验
2. 校验通过后，系统判断填报人是否为牵头部门的正职：
   - **是**：记录状态直接置为 `approved`，审批人/时间自动填入填报人信息，系统备注「填报人为部门正职，自动通过」
   - **否**：记录状态置为 `pending`，推送通知给审批人（牵头部门正职；若无正职则为第一副职）
3. 提交成功后展示成功提示，询问用户「继续登记」或「查看我的记录」

#### 2.1.3 修改重提（预填模式）

当从「我的记录」点击「修改重提」进入本页面时（仅在记录被退回后可用）：

1. 页面标题变更为「修改并重新提交」，并展示原记录的退回原因（只读，醒目提示）
2. 表单所有字段预填入原记录数据（含动态列表的对方参与人员、开拓对象、附件）
3. 填报人、所属部门仍为只读，不可修改
4. 点击「重新提交审批」，校验通过后调用重提接口，记录状态回到 `pending`，原历史审批链保留，追加一条新的「重提」记录到历史链
5. 提交成功后跳转至「我的记录」

#### 2.1.4 重置

点击「重置」按钮，清空当前表单（非预填模式），还原至默认初始状态（填报人/部门不变）。预填模式（修改重提）下，重置还原为原记录数据。

### 2.2 我的记录

**入口**：左侧导航「我的记录」

**权限**：所有登录用户

**页面用途**：查看当前登录用户提交的所有活动记录，支持状态筛选、撤回、修改重提、查看详情。

#### 2.2.1 状态统计卡片

页面顶部展示 4 张统计卡片：

| 卡片 | 统计范围 |
|------|---------|
| 全部 | 当前用户所有记录数 |
| 待审批 | `status = pending` |
| 已通过 | `status = approved` |
| 已退回 | `status = rejected` |

点击卡片即切换下方列表的筛选状态。

#### 2.2.2 记录列表

每条记录展示：
- 活动标题
- 状态标签（待审批 / 已通过 / 已退回 / 草稿）
- 活动时间（`dateStart` 至 `dateEnd`，同一天则只显示一个日期）
- 活动形式
- 牵头部门
- 提交时间

操作按钮（按状态显示）：

| 状态 | 可用操作 |
|------|---------|
| `pending` | 查看详情 |
| `approved` | 查看详情 |
| `rejected` | 查看详情、修改重提 |

**修改重提**：仅在 `rejected` 状态时可用。点击后跳转至活动登记表单（预填模式），见 2.1.3。

#### 2.2.3 详情抽屉

点击「查看详情」，从右侧滑出详情抽屉，展示该记录全部字段信息，分区展示（与表单分区对应）。

额外展示以下审批信息：
- 当前状态
- 提交时间
- 审批人及审批时间（已通过时展示）
- 退回原因（已退回时展示，醒目标注）
- 抄送人员（已通过时展示）
- 历史审批链（见第三章）：完整展示每一轮操作记录

抽屉底部按钮：与列表操作按钮保持一致（按当前状态显示）。

### 2.3 待我审批

**入口**：左侧导航「待我审批」

**权限**：`role = head` 的用户（各部门正职）

**页面用途**：展示需要当前用户审批的活动记录，支持审批通过、退回操作。

#### 2.3.1 记录列表

展示所有 `status = pending` 且牵头部门正职为当前用户的记录。

每条记录展示：
- 活动标题
- 填报人及所属部门
- 活动时间
- 活动形式
- 提交时间

操作按钮：查看详情、审批通过、退回修改

若无待审批记录，展示空状态提示。

#### 2.3.2 审批通过

点击「审批通过」，弹出确认弹窗：

- 展示活动标题（只读，用于确认）
- **自动抄送**：战略发展部正职（固定，不可取消，灰显）
- **可选追加抄送**：从本部门成员中多选，副职默认预选
- 填写审批意见（可选，文本框）

点击「确认通过」后：
1. 记录状态变为 `approved`
2. 抄送列表 = 战略发展部正职 + 所选追加人员（去重）
3. 通过通知系统向填报人发送「审批通过」通知，向所有抄送人发送抄送通知
4. 记录本次操作到审批历史链

#### 2.3.3 退回修改

点击「退回修改」，弹出退回弹窗：

- 退回原因（必填文本框，最多 500 字）

点击「确认退回」后：
1. 记录状态变为 `rejected`，退回原因存入记录
2. 通过通知系统向填报人发送「记录已被退回」通知，附带退回原因
3. 记录本次操作到审批历史链

#### 2.3.4 查看详情抽屉

与「我的记录」的详情抽屉内容相同，抽屉底部额外展示「审批通过」和「退回修改」操作按钮。

| 筛选项 | 组件 | 说明 |

展示当前筛选结果，支持全选/多选。

| 列名 | 字段 | 说明 |

| 填报人 | reporter | — |
| 活动时间 | dateStart / dateEnd | 同一天只显示一个 |
| 活动形式 | activityType | — |
| 规模（人） | scale | — |
| 开拓对象 | targets | 多个时换行展示机构名称 |
| 所属行业 | targets[].industry | 去重后展示 |
| 牵头部门 | leadDept | — |
| 配合部门 | supportDept | 多个以顿号分隔 |
| 我委参与人员 | ourStaff | 多人以顿号分隔 |
| 审批状态 | status | 标签样式 |
| 操作 | — | 「查看详情」按钮 |

表格右上角提供「导出 Excel」按钮。

## 三、审批流程规格

### 3.1 状态定义

| 状态值 | 含义 |
|--------|------|
| `pending` | 待审批，已提交等待审批人处理 |
| `approved` | 已通过 |
| `rejected` | 已退回，填报人可修改重提 |

### 3.2 状态流转

```
[新建提交]
  │
  ├─ 填报人是牵头部门正职 ──→ approved（自动通过）
  │
  └─ 填报人不是正职 ──→ pending
                        │
              ┌─────────┼─────────┐
              ↓         ↓         ↓
         [修改重提]  [审批通过] [退回修改]
              │         │         │
           pending   approved  rejected
              ↑                    │
              └──── [修改重提] ────┘
```

### 3.3 审批人确定规则

审批人 = 牵头部门的负责人，按以下优先级查找：

1. 该部门有正职（`role = head`）→ 正职
2. 该部门无正职 → 该部门副职列表中的第一位

> 若部门既无正职也无副职，提交时前端提示「该部门暂无审批人，请联系管理员」，阻止提交。

### 3.4 自动通过规则

提交时，若填报人姓名 === 牵头部门的正职姓名：
- 记录直接置为 `approved`
- `approver` = 填报人
- `approveTime` = 提交时间
- 系统在审批历史中记录：操作类型 `auto_approved`，备注「填报人为部门正职，自动通过」
- 同样触发抄送通知（自动抄送战略发展部正职）

### 3.5 通知规则

系统通过调用甲方现有通知服务接口发送站内消息/通知，具体接口由甲方提供；本文档仅定义触发时机和消息内容模板。

| 触发事件 | 接收人 | 消息内容模板 |
|---------|--------|------------|
| 初次提交（非自动通过） | 审批人 | 「{填报人} 提交了一条活动记录「{活动标题}」，请审批。」 |
| 修改重提 | 审批人 | 「{填报人} 修改并重新提交了活动记录「{活动标题}」，请审批。」 |
| 审批通过 | 填报人 | 「您提交的活动记录「{活动标题}」已通过审批。」 |
| 审批通过（抄送） | 抄送人员 | 「活动记录「{活动标题}」（填报人：{填报人}）已通过审批，现抄送您知悉。」 |
| 退回修改 | 填报人 | 「您提交的活动记录「{活动标题}」被退回，退回原因：{退回原因}。请修改后重新提交。」 |

## 四、数据模型

### 4.1 实体关系

```
用户（User）           部门（Department）
  ├── 填报 ──→ 活动记录（Activity）←── 隶属 ── 部门
  └── 审批 ──→ 审批历史（ApprovalHistory）←── 归属
                   │
                   └── 活动记录 1:N 开拓对象（Target）
```

### 4.2 活动记录（Activity）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | bigint / UUID | 主键 |
| title | varchar(60) | 活动标题 |
| reporter_id | 外键 → User | 填报人 |
| reporter_dept | varchar | 填报人部门名称（冗余存储，防止部门调整后历史数据变动） |
| contact_id | 外键 → User | 我委对接人 |
| lead_dept | varchar | 牵头部门名称 |
| support_dept | json array | 配合部门名称列表 |
| date_start | date | 活动开始日期 |
| date_end | date | 活动结束日期 |
| activity_type | varchar | 活动形式（枚举） |
| scale | int | 活动规模（人数） |
| our_staff | json array | 我委参与人员姓名列表 |
| our_leaders | json array | 出席领导姓名列表 |
| other_staff | json array | 对方参与人员，每项 `{name, title, contact}` |
| purpose | text | 活动目的及成效（≤1000字） |
| feedback | text | 反馈意见建议（≤1000字） |
| follow_up | text | 后续跟进事项（≤1000字） |
| attachments | json array | 附件列表，每项 `{filename, path, size, mime_type}` |
| remarks | text | 备注（≤500字） |
| status | varchar | `pending` / `approved` / `rejected` |
| submit_time | datetime | 最近一次提交时间 |
| created_at | datetime | 记录创建时间 |
| updated_at | datetime | 记录最后更新时间 |

### 4.3 开拓对象（Target）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | bigint / UUID | 主键 |
| activity_id | 外键 → Activity | 所属活动记录 |
| name | varchar | 机构名称 |
| industry | varchar | 所属行业（枚举） |
| province | varchar | 所在省份 |
| city | varchar | 所在城市 |
| sort_order | int | 显示排序（保持填写顺序） |

### 4.4 审批历史（ApprovalHistory）

每条审批操作独立记录，支持多轮退回-修改-重提的完整链路追溯。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | bigint / UUID | 主键 |
| activity_id | 外键 → Activity | 所属活动记录 |
| round | int | 第几轮审批（从 1 开始，每次重提 +1） |
| action | varchar | 操作类型，枚举：`submitted` / `resubmitted` / `approved` / `auto_approved` / `rejected` |
| operator_id | 外键 → User | 操作人 |
| operator_name | varchar | 操作人姓名（冗余存储） |
| operator_dept | varchar | 操作人部门（冗余存储） |
| comment | text | 审批意见 / 退回原因 |
| cc_list | json array | 抄送人员姓名列表（仅 `approved` 类型有值） |
| operated_at | datetime | 操作时间 |


### 4.5 用户（User）

由甲方 SSO 系统维护，本模块只读。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | — | SSO 用户 ID |
| name | varchar | 姓名 |
| dept | varchar | 所属部门 |
| role | varchar | `head` / `deputy` / `staff` |
| title | varchar | 职务名称（如部长、副部长、干部） |

### 4.6 部门（Department）

由甲方系统维护，本模块只读（或在本模块内做镜像同步）。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | — | 部门 ID |
| name | varchar | 部门名称 |
| head_user_id | 外键 → User | 正职用户 ID，可为空 |
| deputy_user_ids | json array | 副职用户 ID 列表 |

## 五、非功能需求

### 5.1 权限控制

- 所有接口均需验证 SSO token，token 失效时跳转甲方统一登录页
- 「查询与统计」接口需在后端校验调用者的角色权限（`role = head` 或 `dept = 战略发展部`），前端隐藏菜单项不可作为唯一防线
- 「待我审批」操作接口（通过/退回）需在后端校验操作人是否为目标记录牵头部门的正职
